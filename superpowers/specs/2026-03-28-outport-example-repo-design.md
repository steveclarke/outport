# outport-example — Reference Repository Design

A standalone example repository demonstrating Outport's port orchestration, cross-service env wiring, and devstack integration in a realistic multi-service setup. All services are bare Node/Express — no frameworks, no build steps. The focus is entirely on the wiring.

## Goal

Help developers understand Outport by showing, not telling. Clone the repo, run `outport up`, and see every service get the right ports, hostnames, and env vars wired together automatically.

## Services

| Service | Hostname | Port env var | Description |
|---------|----------|-------------|-------------|
| `api` | `api.example.test` | `API_PORT` | Backend API — returns JSON with all its env vars |
| `app` | `example.test` | `APP_PORT` | Primary frontend — HTML page showing its env vars |
| `admin` | `admin.example.test` | `ADMIN_PORT` | Admin frontend — same pattern, different identity |
| `docs` | `docs.example.test` | `DOCS_PORT` | Docs site — minimal, shows the pattern scales |

Each service is a ~30-line Express server:
- Loads `dotenv`
- Starts on its port env var
- API: serves JSON at `/` listing all Outport-managed env vars, plus `/health` for process-compose
- Frontends/docs: serves an HTML page showing service name, port, hostname, API URL, and other env vars it received. Styled just enough to not look broken.

## outport.yml

```yaml
name: example

services:
  api:
    env_var: API_PORT
    hostname: api.example
    env_file:
      - api/.env
  app:
    env_var: APP_PORT
    hostname: example
    env_file:
      - app/.env
  admin:
    env_var: ADMIN_PORT
    hostname: admin.example
    env_file:
      - admin/.env
  docs:
    env_var: DOCS_PORT
    hostname: docs.example
    env_file:
      - docs/.env

computed:
  # Frontend → API wiring (server-to-server, use :direct)
  API_URL:
    value: "${api.url:direct}"
    env_file:
      - app/.env
      - admin/.env

  # Frontend → API wiring (browser-facing, for display)
  API_HOSTNAME:
    value: "${api.url}"
    env_file:
      - app/.env
      - admin/.env

  # API → Frontend wiring (browser-facing, for CORS)
  CORS_ORIGINS:
    value: "${app.url},${admin.url}"
    env_file: api/.env

  # Docker Compose isolation per worktree
  COMPOSE_PROJECT_NAME:
    value: "${project_name}${instance:+-${instance}}"
    env_file: docker/.env

  # Process-compose integration
  HEALTH_CHECK_URL:
    value: "${api.url:direct}/health"
    env_file: .pc_env

  PC_SOCKET_PATH:
    value: "/tmp/process-compose-${project_name}${instance:+-${instance}}.sock"
    env_file: .pc_env

  # Bruno API testing (server-to-server)
  BRUNO_API_URL:
    value: "${api.url:direct}"
    env_file: bruno/.env
```

### Key features demonstrated

- **Multi-service port allocation** — 4 services with `.test` hostnames
- **Cross-service env wiring** — API port written to frontend env files, CORS origins computed from frontend URLs
- **`url` vs `url:direct`** — browser-facing `.test` URLs vs server-to-server `localhost:port`
- **Docker Compose isolation** — `COMPOSE_PROJECT_NAME` with `${instance}` expansion for worktree safety
- **Process-compose integration** — `HEALTH_CHECK_URL` and `PC_SOCKET_PATH` as computed values
- **API testing tool integration** — Bruno URLs wired from env vars
- **Template expansion** — `${instance:+-${instance}}` pattern

## process-compose.yml

```yaml
version: "0.5"

shell:
  shell_command: "bash"
  shell_argument: "-lc"

processes:
  api:
    command: bin/start-api
    readiness_probe:
      exec:
        command: "curl -sf ${HEALTH_CHECK_URL}"
      initial_delay_seconds: 1
      period_seconds: 3
      failure_threshold: 5

  app:
    command: bin/start-app
    depends_on:
      api:
        condition: process_healthy

  admin:
    command: bin/start-admin
    depends_on:
      api:
        condition: process_healthy

  docs:
    command: bin/start-docs
```

Demonstrates:
- Health checks using Outport-computed `HEALTH_CHECK_URL`
- Dependency ordering (frontends wait for healthy API)
- UDS socket via `PC_SOCKET_PATH` in `.pc_env` (auto-loaded by process-compose)
- Each process delegates to a `bin/` script

## bin/ scripts

### bin/start-api, bin/start-app, bin/start-admin, bin/start-docs

Each is a 2-3 line shell script:

```bash
#!/usr/bin/env bash
cd "$(dirname "$0")/../api" && npm start
```

These are what process-compose calls. They also work standalone — run them in separate terminals for the manual approach.

### bin/test-api

Curl script that sources env vars and hits each service:

```bash
#!/usr/bin/env bash
set -e

# Source the API env to get the URL
source api/.env

echo "=== API Status ==="
curl -s "http://localhost:${API_PORT}/"
echo

echo "=== App Status ==="
source app/.env
curl -s "http://localhost:${APP_PORT}/"
echo

echo "=== Admin Status ==="
source admin/.env
curl -s "http://localhost:${ADMIN_PORT}/"
echo
```

## docker/compose.yml

A placeholder Docker Compose file that demonstrates the `COMPOSE_PROJECT_NAME` pattern. Does not need Docker to be installed for the rest of the example to work.

```yaml
# This file demonstrates how Outport isolates Docker Compose
# across worktrees via COMPOSE_PROJECT_NAME.
#
# The .env file in this directory is generated by `outport up`
# and contains COMPOSE_PROJECT_NAME=example (or example-xbjf
# for worktrees).
#
# If you have Docker installed, `docker compose up` works.
# If not, the rest of the example still runs fine.

services:
  db:
    image: postgres:17
    environment:
      POSTGRES_PASSWORD: postgres
```

## Root package.json

Workspace-style package.json for installing all subdirectories at once:

```json
{
  "name": "outport-example",
  "private": true,
  "workspaces": ["api", "app", "admin", "docs"],
  "scripts": {
    "setup": "npm install"
  }
}
```

`npm install` at the root installs all four services' dependencies.

## README.md

Sections:

1. **What is this?** — One paragraph: a reference repo showing how Outport orchestrates ports, hostnames, and env vars across a multi-service dev environment.

2. **Prerequisites** — Node.js, Outport. Optional: process-compose, Bruno, Docker.

3. **Quick start**
   ```

   npm install
   ```

5. **What just happened?** — Walk through what `outport up` did: allocated ports, computed cross-service URLs, wrote everything to `.env` files. Show a before/after of `api/.env`.

6. **Explore** — Visit each service URL, run `bin/test-api`, open the Bruno collection.

7. **Try worktrees** — Create a git worktree, run `outport up` there, see different ports and `COMPOSE_PROJECT_NAME`. This is the isolation proof.

8. **Learn more** — Link to Outport docs, the `outport.yml` reference, and the Outport dashboard.
